Security & control

Automation with visibility and control.

The parts of support that can be automated, lockouts and password resets, run through a server agent inside your network. Every job is signed, outbound-only, and written back to the ticket, so nothing happens without a record.

How a job runs, safely.

The server agent pulls approved work over an outbound connection and reports each step back. It never opens an inbound port.

The AI proposes an approved action

Only actions your policy allows, like an account unlock or password reset, are queued.

The agent picks it up outbound

The server agent reaches out to the queue. No inbound ports, no exposed listeners.

It runs inside your network

Signed jobs execute against your directory, optionally restricted to an allowlist.

The result posts to the ticket

Every action becomes a visible, auditable record on the ticket it belongs to.

Controls, not promises

Guardrails on every automated action.

Automation is only useful if you can trust it. These controls keep the server agent inside clear boundaries and make every action reviewable.

  • Outbound connections only. No inbound ports.
  • Account actions can start in dry-run mode.
  • Optional allowlist restricts which accounts can be acted on.
  • Password resets force a change at next sign-in.
  • Temporary passwords are delivered once, never persisted in logs.
  • Ticket updates create a visible record of every action.
  • Admin-only controls protect ticket management and device actions.

The principles behind it.

Least exposure

Outbound-only by design. The agent never opens your network to inbound traffic.

Everything auditable

Each automated step is written back to the ticket as a visible, permanent record.

Bounded by policy

Dry-run mode and allowlists let you introduce automation at the pace you're comfortable with.
